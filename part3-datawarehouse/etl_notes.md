## ETL Decisions
### Decision 1 — Date Format Standardization
The raw dataset had all dates written in DD-MM-YYYY format, for example 29-08-2023. While that's readable for humans, it doesn't work well when you're trying to sort records or run analytical queries — the system doesn't naturally know that's a date. To fix this, every date was converted to the YYYY-MM-DD format before being loaded into the dim_date table. This small change makes a big difference — it keeps things consistent and makes time-based analysis like monthly or yearly trends much easier to run.
### Decision 2 — Category Normalization
The dataset had the same categories written in different ways — "electronics" and "Electronics", or "Grocery" and "Groceries" were all floating around as separate values. When you try to group or total data by category, this causes wrong results because the system treats them as different things. To sort this out, all category values were cleaned up and brought into one consistent format — "Electronics", "Clothing", "Groceries" — before being loaded into the dim_product table.
### Decision 3 — Handling Missing Store City Values
Some records were missing the store_city field — stores like "Mumbai Central" and "Delhi South" had no city listed against them. This would have created gaps in any location-based reporting or analysis. Since the city could be figured out directly from the store name, the missing values were filled in logically — "Mumbai Central" was mapped to "Mumbai", "Delhi South" to "Delhi", and so on. This kept the dimensional data complete and reliable without having to drop any records.
